# 80_backlog — Backlog / Roadmap (без переписывания пол‑проекта)

## 0) Зачем backlog здесь
Backlog нужен не “для красоты”, а чтобы:
- фиксировать идеи, которые **логично вытекают** из архитектуры,
- не превращать MVP в бесконечную стройку,
- иметь план расширения каналов/фич без распила ядра.

Формат: **пакеты работ** + критерии готовности + оценка риска.

---

## 1) MVP v0 (считаем “готово”)
### SB-MVP-00
**Состав:**
- Resolve `/v1/mc/resolve` (hit/fallback_catalog)
- Redirect `/t/{slug}` (302)
- content_map CRUD + import/export (admin token)
- inbound_events dedup + click logging
- docker-compose + alembic baseline
- smoke tests + docs

**DoD:**
- 10 кампаний через `campaign:*` работают
- CTR и клики логируются
- reset в dev работает, prod миграции не ломают

---

## 2) Каналы и интеграции (расширение без боли)

### SB-CH-01: TikTok “подключился в ManyChat” (если это вообще случится)
**Суть:** добавляем `tt` как channel без изменений ядра.
- ManyChat flow клонируется с IG
- `sb_channel = tt`
- `content_ref` остаётся `campaign:*`

**DoD:** 3 кампании в tt работают как в ig.

**Риск:** geo/правила платформ.

### SB-CH-02: Прямой Instagram (без ManyChat) — *опционально и позже*
**Суть:** прямые вебхуки/DM через Meta API.
**Причина:** контроль и снижение зависимости от ManyChat.
**Но:** комплаенс и review боль.
**Риск высокий** → отдельный проект.

### SB-CH-03: WhatsApp / Telegram DM / Email / Web chat (будущее)
**Суть:** SocialBridge как “маршрутизатор ссылок”.
**Подход:** новый адаптер + новый normalizer, use-case тот же.

## 3) Shortlink 2.0 (когда кликов станет много)
### SB-LINK-01: Агрегации
**Проблема:** `sb_click_event` растёт.
**Решение:** `sb_daily_aggregate` и/или materialized views.

**DoD:**
- отчёт по кликам работает на агрегатах
- retention raw click_events сокращён (например до 90 дней)

### SB-LINK-02: UTM/параметры
**Суть:** поддержка `?utm_source=...` на shortlink.
**Но:** аккуратно с PII.
**DoD:** UTM сохраняются в `meta` на click_event или агрегатах.

### SB-LINK-03: Защита от перебора slug
**Опции:**
- Cloudflare rate limit
- honeypot slug
- random suffix (например `dress001-x7k`)
- “словарь” slug только для топ‑кампаний

## 4) Resolve 2.0 (умнее, но без магии)
### SB-RES-01: Dynamic shortlinks (payload без content_map)
**Идея:** если в тексте пришёл `LOOK_...` или код товара — не требовать записи в маппинге.

**Вариант A:** `GET /t/_/{start_param}` (валидированный)
**Вариант B:** `/r/{hash}` где hash → start_param в cache

**Риск:** сканирование payload, нужен строгий валидатор.

### SB-RES-02: “Smart” поиск по тексту (опционально)
**Идея:** пользователь пишет “тот белый костюм с фото”.
**Подход:**
- сначала строгие правила (regex/словарь)
- потом LLM как последняя линия, но только если есть каталог/индекс.

**Риск:** расползание. Держать отдельно.

### SB-RES-03: Мульти-язычные ответы
**Идея:** reply_text в зависимости от locale.
**Подход:** шаблоны в config/DB, не в коде.

## 8) Корреляция с SIS (повышение точности аналитики)
### SB-SIS-01: Проброс campaign в SIS events
**Идея:** чтобы в SIS аналитике видеть источник кампании.
**Как:**
- либо расширить payload (опасно из-за 64 символов)
- либо через параметр в deeplink (Telegram start не поддерживает второй параметр)
- либо через “intermediate” экран в SocialBridge (HTML) с session id (не хотелось бы)

**Реалистичный подход:**
- хранить кампанию в SocialBridge по slug и коррелировать с событиями SIS через временное окно и user_id (если доступно)
- или не лезть в SIS вообще и жить по click KPI

**Риск:** интеграция с SIS и данные, лучше отдельной задачей.

## 9) Надёжность (когда начнётся “настоящая жизнь”)
### SB-REL-01: Redis cache slug→start_param
Чтобы редирект был очень быстрым и DB не страдала.

### SB-REL-02: Очередь для логов (опционально)
Если resolve должен отвечать мгновенно, а логировать можно асинхронно:
- писать события в очередь
- worker пишет в БД

Но это усложнение. Делать только при необходимости.

---

## 11) Принцип расширения (главное правило)
Новый канал/интеграция не должна менять:
- таблицы MVP,
- use-case resolve/redirect,
- контракты API.

**Добавляем только:**
- новый normalizer (adapters)
- новые content_ref соглашения
- (опционально) новые поля в meta

Если приходится “переписывать ядро” — значит, мы нарушили архитектуру. Или кто-то решил поиграть в SIS 2.0.
