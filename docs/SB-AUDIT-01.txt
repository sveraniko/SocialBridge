SB-AUDIT-01 — Repository Audit Report
1) Implemented features (what is реально в репо)
API endpoints are present for all requested surfaces:

GET /health and GET /ready (with DB probe and 503 on failure).

POST /v1/mc/resolve with normalized payload, channel validation, and response fields reply_text/url/start_param/slug/tag/result.

GET /t/{slug} with click logging + 302 redirect target from service logic.

Admin content-map endpoints: list/upsert/import/export/disable under /v1/admin/content-map/*.

POST /v1/admin/resolve-preview exists and intentionally does not persist inbound event log (persist_inbound_event=False).

Auth controls exist:

Admin token enforced on /v1/admin/* (403 via HTTPException if invalid).

ManyChat token check is conditional (MC_TOKEN_REQUIRED).

Resolve behavior implemented per core flow:

Lookup by (channel, content_ref) → hit.

On miss, parse payload from text → dynamic mapping in generic channel → fallback_payload.

Else → catalog slug/result fallback_catalog.

Inbound dedup/log record is written with payload_hash and resolution metadata.

Redirect behavior implemented with miss-safe fallback:

If slug not found, still logs click (meta.miss=true) and redirects to bot catalog deeplink (no start).

Data model + migration baseline are present and aligned in code:

sb_content_map, sb_inbound_event, sb_click_event with checks/uniques/indexes in Alembic baseline and SQLAlchemy models.

Operator scripts exist for required workflows:

Import/upsert/resolve-preview admin helpers and registry CSV→JSON converter are implemented as CLI scripts.

Seed/registry assets are present:

Example campaign registry CSV and pre-generated content map JSON exist and are consistent in content shape for import flow.

2) Missing / partial features (severity-tagged)
Blocker
Environment bootstrap reproducibility risk: repo relies on docker compose for local runbook, but current environment had no docker binary, and pip install is blocked by network/proxy, preventing full operator simulation in this audit environment.

This is an audit environment blocker, not necessarily a repo code blocker, but it blocks “no-touch runtime validation” right now.

Important
API contract mismatch: list default/max pagination differs from docs

Docs: default limit=200, max 1000; code: default 50, max 500.

API contract mismatch: upsert response lacks result=created|updated

Docs specify {"item": ..., "result": "created|updated"}; code returns only {"item": ...}.

API contract mismatch: export filtering not implemented

Docs mention optional query filters on export; endpoint currently ignores filters and returns all records.

Runbook drift: service naming/components

Runbook references db, optional redis/proxy, but compose defines postgres + api only. Operationally fine, but docs and compose naming diverge.

Nice-to-have
No explicit admin audit trail table (sb_admin_audit) yet (documented as optional; useful before broader operator access).

Error response consistency is mostly centralized, but some endpoint-local JSONResponse paths omit request_id on manual 400 branches (e.g., missing channel in resolve).

3) Stubs / placeholders / landmines scan
No TODO/FIXME/NotImplementedError stubs in application logic found by scan.

Only literal pass is SQLAlchemy declarative base class (normal, not a stub risk).

Potential landmine identified: dynamic fallback mapping writes DB rows (dyn:*) during resolve fallback. This is intentional behavior but can grow mapping table over time if traffic has many unique payloads; needs ops watch/retention strategy.
